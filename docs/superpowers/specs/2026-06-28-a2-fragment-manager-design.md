# A2.1 Fragment Manager Design

## Status

Draft for reviewed implementation planning.

## Context

Multipass A1 lets a user Activate a live AgentDNA record, save a durable public profile, and share `/multipass/{slug}`. A2 claim/manage is already partially implemented: source-owner wallet claims, manual review claims, admin approval, scoped manager sessions, CSRF protection, logout, and allowlisted public profile edits.

A2.1 should make claimed profiles useful by letting an approved manager add and maintain safe public fragments. Fragments are the real identity graph surface: wallets, social accounts, AgentDNA/Cred references, endpoint metadata, receipt references, and other evidence. The first manager should be deliberately narrow so it cannot become a backdoor for reputation edits, private data leaks, or live authority changes.

## Goal

Ship a safe fragment manager for claimed saved Multipass records.

An approved manager can create, update, revoke, and review allowed public fragments. Public profile readers can fetch the updated fragment list through the existing `GET /api/multipass/{id}/fragments` route. Every fragment mutation is validated, audited, and reflected in the change log.

## Non-goals

- No Cred score edits.
- No direct reputation or trust-tier changes.
- No private credentials, secrets, API keys, memory, or live signer data.
- No live route authority updates.
- No custody transfer.
- No generalized admin console.
- No paid endpoint enforcement in this phase.
- No automatic external account verification beyond explicitly defined safe checks.

## Current baseline

Existing A2 code already provides the authority layer this feature should reuse:

- `POST /api/multipass/{id}/claim/nonce`
- `POST /api/multipass/{id}/claim/verify`
- `POST /api/admin/multipass/{id}/claims/{claim_id}/approve`
- `PATCH /api/multipass/{id}/profile`
- `POST /api/multipass/{id}/session/logout`
- manager session cookie `multipass_manager`
- CSRF header `x-csrf-token`
- change log entries
- audit events in the saved-record store

This means A2.1 should not invent a new auth model. It should add fragment management behind the same manager session validation used by public profile edits.

## Recommended approach

Build the fragment manager as a small extension of saved records, API routes, and web UI.

### Option A: Saved-record fragment manager, recommended

Add fragment mutation methods to `createSqliteSavedRecords`, expose manager-only API routes, then add a claimed-profile UI section.

Benefits:

- Reuses existing sessions, CSRF, claim status, and SQLite persistence.
- Keeps implementation small and testable.
- Makes saved public profiles immediately more useful.
- Avoids premature contract or payment-layer complexity.

Tradeoff: fragments are server-side saved-record data for now, not a live onchain registry.

### Option B: Contract-first fragment registry

Design and deploy an upgradeable fragment registry before web editing.

Benefits: stronger durability story later.

Tradeoff: too slow for the current product need, and it risks locking the schema before the product surface proves itself.

### Option C: Static fixture fragments only

Add more hardcoded examples without manager editing.

Benefits: fastest display polish.

Tradeoff: does not advance claim/manage usefulness.

Recommendation: Option A.

## Fragment scope for A2.1

A2.1 should allow only public, low-risk fragment classes.

### Allowed fragment types

1. `wallet`
   - Public wallet reference.
   - Default status: `pending`.
   - Default assurance: `self_attested` unless later verified by signature or contract read.
   - Transfer policy: `reverify_on_transfer`.

2. `social`
   - Public social route such as X profile URL or handle.
   - Default status: `pending`.
   - Default assurance: `self_attested`.
   - Transfer policy: `reverify_on_transfer`.

3. `endpoint`
   - Public endpoint metadata such as profile JSON, agent card, Bankr x402 Cloud planning URL, or public service URL.
   - Default status: `pending`.
   - Default assurance: `self_attested`.
   - Transfer policy: `pause_on_transfer`.
   - Must use `https` URLs only.

4. `standard_ref`
   - Public standards reference such as ERC-8004, ERC-8217, ERC-8126, ERC-8257, ERC-8183, ERC-721T, or ERC-8048.
   - Default status: `pending` unless created by a trusted resolver/importer.
   - Default assurance: `self_attested`.
   - Transfer policy depends on the standard reference, with `reverify_on_transfer` as default.

5. `attestation`
   - Public statement or proof pointer from an owner or known issuer.
   - Default status: `pending`.
   - Default assurance: `self_attested`.
   - Transfer policy: `historical_on_transfer` or `reverify_on_transfer`.

### Import-only fragment types

`risk_summary` is read-only in A2.1. It may appear when generated by trusted Helixa import code, but managers cannot create, update, or revoke risk-summary fragments. This prevents public fragment editing from becoming a Cred or reputation authority path.

### Explicitly blocked fragment types for manager creation

- `receipt`
- `work_record`
- `custody_record`
- `verification_result`
- `tool_manifest`
- `collection`
- `project`
- `domain`
- `risk_summary`

These can be added later with stronger issuer, payment, or verification flows.

## Data model

The existing `identity-fragment.schema.json` remains the source of truth. A2.1 should not add new top-level fragment fields unless tests update the schema first.

Saved-record storage should persist fragments in the existing `saved_records.fragments_json` bundle for the first implementation. Each mutation should rewrite the validated fragments array in a single SQLite transaction.

A manager-created fragment should include:

- `schema_version: "0.1.0"`
- generated `fragment_id`
- parent `multipass_id`
- allowed `fragment_type`
- normalized `status`
- normalized `assurance_level`
- `visibility: "public"`
- normalized `transfer_policy`
- `source.source_type: "owner_submission"`
- `source.source_id` generated from the fragment id
- `source.issuer: null` for manager-created fragments, so public fragment reads do not expose the manager session wallet
- `source.observed_at`
- safe `source.reference_url` when provided
- `public_value` with length limit and script/content sanitization
- optional `proof_reference`
- `created_at`
- `updated_at`
- optional `endpoint_ref` for endpoint fragments

The actor wallet from the manager session should be recorded only in the private audit event and change metadata, not in the public fragment document. If a future issuer flow needs a public issuer, it must be created by a trusted importer or explicit issuer-attestation path, not by the generic manager editor.

Every fragment mutation must also refresh the saved profile bundle:

- regenerate `profile.public_fragments` from the public fragments array using the summary shape in `multipass-profile.schema.json`
- update `profile.updated_at` to the mutation timestamp
- keep `GET /api/multipass/{id}` and `GET /api/multipass/{id}/fragments` consistent after create, update, or revoke

## API design

All write routes require the same trusted-origin, manager-cookie, and CSRF checks as `PATCH /api/multipass/{id}/profile`.

### List fragments

Existing route:

`GET /api/multipass/{id}/fragments`

Returns public fragments only. It should continue to work for saved records and fixture records.

### Create fragment

`POST /api/multipass/{id}/fragments`

Body shape:

```json
{
  "fragment_type": "wallet",
  "public_value": "0x...",
  "reference_url": "https://...",
  "proof_reference": "optional public proof pointer",
  "endpoint_ref": {
    "endpoint_id": "profile-json",
    "url": "https://helixa.xyz/multipass-api/api/multipass/bendr-2-1",
    "protocol": "api"
  },
  "transfer_policy": "reverify_on_transfer"
}
```

`reference_url` is a request-only convenience field. The API must map it to `source.reference_url` before validating the fragment with `assertIdentityFragment`. No top-level `reference_url` field may be stored because the identity-fragment schema does not allow it.

Response:

```json
{
  "schema_version": "0.1.0",
  "fragment": {},
  "fragments": { "schema_version": "0.1.0", "multipass_id": "...", "fragments": [] },
  "profile": {}
}
```

### Update fragment

`PATCH /api/multipass/{id}/fragments/{fragment_id}`

Allowed updates:

- `public_value`
- `reference_url`, request-only and mapped to `source.reference_url`
- `proof_reference`
- `endpoint_ref`
- `status`, limited to `pending`, `stale`, `revoked`, or `disputed` for manager-created fragments
- `transfer_policy`, only within safe values

Blocked updates:

- `multipass_id`
- `fragment_id`
- `schema_version`
- `source.source_type`
- `assurance_level`, except lowering it to `unverified` or `self_attested`
- `visibility`, except keeping it `public`
- trusted imported fragments unless marked editable

For A2.1, only manager-created fragments can be updated through the public manager UI or generic manager API. Imported Helixa fragments, Cred/risk summaries, and trusted resolver fragments are read-only unless a later spec defines a narrower importer-owned edit path.

### Revoke fragment

`POST /api/multipass/{id}/fragments/{fragment_id}/revoke`

Marks the fragment `revoked`, sets `revoked_at`, appends a change log entry, and keeps history visible unless later policy changes.

## Store design

Add methods to `createSqliteSavedRecords`:

- `createPublicFragment(identifier, input, context)`
- `updatePublicFragment(identifier, fragmentId, patch, context)`
- `revokePublicFragment(identifier, fragmentId, context)`
- `getFragment(identifier, fragmentId)`

The store should:

- Resolve slug or `multipass_id`.
- Load current saved bundle.
- Validate fragment mutations with `assertIdentityFragment`.
- Reject private, hidden, or gated visibility for manager-created fragments.
- Reject unsafe URLs and script-like content.
- Preserve imported fragments unless explicitly editable by policy.
- Regenerate `profile.public_fragments` and `profile.updated_at` from the updated fragment set.
- Write fragments, refreshed profile JSON, change log, and audit event in one transaction.

## Web UI design

Add a `Fragment manager` section under the existing claim management panel, visible only when `claimCsrfToken` exists.

The first UI must include:

- existing fragment list with type, status, assurance, transfer policy, and public value
- create form with fragment type, public value, optional reference URL, optional proof reference
- endpoint fields shown only for endpoint fragments
- revoke button for manager-created fragments
- edit form for manager-created fragments covering the same safe update fields exposed by the API

The UI does not need bulk editing, drag-and-drop ordering, advanced verification workflows, or a full admin console in A2.1.

Copy should be explicit:

- “Public fragments only.”
- “Does not edit Cred score.”
- “Does not grant tools, custody, credentials, or live authority.”

## Security and privacy rules

- Require manager session cookie.
- Require CSRF token.
- Require trusted origin.
- Use existing same-site cookie behavior.
- Use strict allowlists for fragment types and editable fields.
- Allow only `https` reference and endpoint URLs.
- Normalize wallet addresses to lowercase for storage, but display can preserve compact formatting later.
- Strip empty fields.
- Enforce length limits.
- Reject `javascript:`, `data:`, `file:`, raw HTML tags, and suspicious script strings.
- Do not expose non-public fragments through public reads.
- Do not let manager-created data claim platform verification.
- Do not let manager-created data change `cred_summary`, Cred tier, owner custody, session state, or standards compatibility summary.

## API error semantics

Use the existing structured error envelope. Expected errors:

- invalid or blocked fragment type: `400 invalid_request`
- unsafe URL or unsafe public content: `400 invalid_request`
- missing fragment: `404 not_found`
- imported or non-editable fragment mutation: `403 forbidden`
- missing manager session cookie: `401 unauthorized`
- expired, wrong-scope, revoked, or bad-CSRF manager session: `403 forbidden`
- no-op update with no accepted changes: `400 invalid_request`

## Testing plan

API and store tests:

- manager can create an allowed public wallet fragment.
- manager can create an endpoint fragment with safe `https` endpoint metadata.
- manager cannot create blocked fragment types.
- manager cannot create gated/private/hidden fragments.
- manager cannot create or edit Cred score/risk summary directly.
- unauthenticated requests fail.
- missing or bad CSRF fails.
- wrong session scope fails.
- unsafe URLs fail.
- fragment mutations append change log and audit events.
- revoked fragments remain historical and visible as revoked public records.
- saved-record public fragment reads return updated fragments.

Web tests:

- claimed session renders fragment manager.
- unclaimed saved profile does not render fragment manager.
- create fragment form calls API and re-renders updated fragments.
- revoke action calls API and updates status.
- edit action calls API and re-renders updated safe fields.
- wrong-wallet or expired session errors are human-readable.
- public copy avoids forbidden language and overclaims.

Verification gates:

- `pnpm test`
- `MULTIPASS_BASE=/multipass/ pnpm --filter @helixa/multipass-web build`
- `git diff --check`
- wording scan for forbidden public labels and trust overclaims

## Rollout plan

1. Implement store methods and tests.
2. Implement API routes and tests.
3. Implement web client helpers and tests.
4. Add UI section behind existing claimed session state.
5. Run full verification.
6. Deploy to `https://helixa.xyz/multipass/` only after tests pass and the work is reviewed.

## Decisions for A2.1

- `social` supports both handles and `https` URLs, with clear display labels and no implied platform verification.
- Manager-created wallet fragments remain `self_attested` until signed or imported by a trusted resolver. A manager session alone does not make a wallet fragment verified.
- Bankr x402 endpoint metadata is stored as an `endpoint` fragment first. Updating `x402Manifest` belongs to A2.2.
