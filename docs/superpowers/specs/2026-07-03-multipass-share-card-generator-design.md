# Multipass Share Card Generator Design

## Decision

Build option 1: a checked-in static share-card generator for Multipass agent previews.

The generator becomes the source-controlled path for updating crawler pages and 1200x630 preview images under `apps/web/public/share/`. The app should no longer rely on hand-maintained cache-version maps or manually edited share HTML when an agent visual changes.

## Product goal

When a Multipass profile is shared to Telegram, X, Discord, or similar crawlers, the preview should show the current public visual for that Multipass profile.

A maintainer should be able to regenerate the share preview assets from existing Multipass demo/profile data, run tests, and know whether the static assets match the current agent visual sources.

## Scope

### In this slice

- Add a static generator for every numeric static agent card in `STATIC_DEMO_DATA.agentCards`; at the time of this design, those cards are `0`, `1`, and `81`.
- Generate per-agent crawler HTML files under `apps/web/public/share/<tokenId>/index.html`.
- Generate per-agent preview images under `apps/web/public/share/<tokenId>.png` and `.jpg`.
- Generate a checked-in share-card manifest consumed by the web app, for example `apps/web/src/generated-share-cards.js`.
- Compute share cache versions from the final generated JPEG bytes instead of a hand-coded version map. This handles upstream visual content changes even when the visual URL stays the same.
- Reuse one shared JavaScript helper for app share URLs and generated HTML image URLs.
- Add tests that fail when share metadata, versioning, or generated image provenance drifts from `STATIC_DEMO_DATA`.
- Keep human routing behavior the same: crawlers see static metadata; humans are redirected/opened into `/multipass/?agent=<tokenId>`.

### Explicit non-goals

- Do not add a dynamic OG image service.
- Do not introduce paid external image generation.
- Do not call private APIs or use secrets.
- Do not rebuild every saved Multipass profile yet.
- Do not change the public profile layout, claim flow, Tool Registry, or manager APIs.
- Do not make share previews dependent on a live server at request time.

## User experience

### Maintainer flow

A maintainer runs one command from the repo root:

```bash
pnpm generate:share-cards
```

The command reads the known static agent cards, writes deterministic static share HTML, regenerates preview PNG/JPEG files, and writes the checked-in manifest. If the rendered preview image changes, the version query changes automatically so Telegram/X crawlers request a fresh image URL.

For verification and CI-like checks, a maintainer can run:

```bash
pnpm generate:share-cards --check
```

`--check` renders into temporary output and exits non-zero if any checked-in generated file would change.

### End-user flow

The share icon still shares a crawler-friendly URL like:

```text
https://helixa.xyz/multipass/share/81/?v=<version>
```

Crawlers read metadata from that page. Humans opening the same URL are sent to:

```text
https://helixa.xyz/multipass/?agent=81
```

## Architecture

### Shared share-card helper

Create a small web module, for example `apps/web/src/share-cards.js`, with pure helpers:

- `getAgentSharePath({ tokenId, version })`
- `getAgentShareImageUrl({ tokenId, version, origin })`
- `getAgentSharePageUrl({ tokenId, version, origin })`
- `getAgentShareCard(tokenId, manifest)`

The path helper emits exactly `/multipass/share/<tokenId>/?v=<version>` when a generated manifest version exists. The image helper emits exactly `https://helixa.xyz/multipass/share/<tokenId>.jpg?v=<version>` when the production origin is passed. The page URL helper emits exactly `https://helixa.xyz/multipass/share/<tokenId>/?v=<version>` when the production origin is passed. The canonical crawler URL stays queryless: `https://helixa.xyz/multipass/share/<tokenId>/`.

Missing-manifest behavior is explicit: `getAgentShareCard(tokenId, manifest)` returns `null` when no generated card exists, and `getAgentSharePath()`/image/page helpers return `null` when called without a valid token ID and version. The aura card should render no static share button for non-generated profiles rather than pointing to a broken `/multipass/share/<id>/` route. Existing non-static profile sharing surfaces can keep using their current profile URLs.

The version is generated by the generator from the final JPEG bytes, using a short SHA-256 prefix. The helper only formats the version; it does not compute hashes or fetch images in the browser.

The app uses these helpers plus the generated manifest when rendering the aura share button. The generator uses the same helpers when writing HTML metadata. This removes the current one-off `AGENT_SHARE_CACHE_VERSIONS` map.

### Generator script

Add a repo script, for example `apps/web/scripts/generate-share-cards.mjs`, exposed from the root package as `pnpm generate:share-cards`.

The generator imports `STATIC_DEMO_DATA` and builds a catalog of every numeric token card. For each card, it writes:

- `apps/web/public/share/<tokenId>/index.html`
- `apps/web/public/share/<tokenId>.png`
- `apps/web/public/share/<tokenId>.jpg`

After image generation, it computes a short SHA-256 prefix from each final JPEG and writes a checked-in manifest, for example:

```js
export const GENERATED_SHARE_CARDS = {
  '81': {
    tokenId: '81',
    version: 'ab052a87',
    visualSource: 'https://assets.bueno.art/images/3b04f823-b7a8-4965-b61e-8fe8a5d82bde/default/2432',
  },
};
```

The generated HTML includes:

- canonical share URL without query,
- `og:url`,
- `og:title`,
- `og:description`,
- `og:image` with the generated manifest version query,
- `twitter:image` with the same generated manifest version query,
- `multipass:visual-source` when a public visual URL exists,
- crawler-safe fallback card markup,
- script redirect for human users.

For image rendering, use one explicit local path:

1. Build an SVG string for each 1200x630 preview card from public card fields.
2. Fetch the public `visual.imageUrl` with a normal browser-like user agent when it exists.
3. Embed the fetched visual as a base64 data URI inside the SVG so rasterization never depends on remote network access.
4. If the visual fetch fails, embed a deterministic initials fallback in the SVG and print a warning naming the token ID.
5. Rasterize SVG to PNG and JPEG with the local `ffmpeg` binary using its `svg_pipe` demuxer and `svg` decoder backed by librsvg, which are already available on the deployment host.

The script must check the exact rasterization path before writing output:

```bash
command -v ffmpeg
ffmpeg -hide_banner -formats | grep ' D .*svg_pipe'
ffmpeg -hide_banner -codecs | grep ' D.V.*svg'
```

If any check fails, or if rasterization fails, exit non-zero and do not silently leave old images in place.

Generation must be all-or-nothing across every generated artifact, not just images. The script should first render all SVGs/images, compute all JPEG hashes, build all HTML pages, and build the manifest under a temporary output root such as `apps/web/public/share/.generated-tmp/`. Only after every card succeeds should it replace final files for `apps/web/public/share/<tokenId>/index.html`, `apps/web/public/share/<tokenId>.png`, `apps/web/public/share/<tokenId>.jpg`, and `apps/web/src/generated-share-cards.js`. The temporary directory must be removed on success and ignored or removed on failure.

### Tests

Extend existing share preview tests so they prove:

- every numeric static agent card in `STATIC_DEMO_DATA.agentCards` has a share HTML route,
- every generated card has `.png` and `.jpg` assets at 1200x630,
- generated manifest versions match short hashes of the final JPEG files,
- generated HTML image URLs match the shared helper output and manifest version,
- generated HTML includes the current `visual.imageUrl` from `STATIC_DEMO_DATA` when present,
- generated HTML omits `multipass:visual-source` when no visual URL exists,
- Quigbot is not using the stale Agent Aura JPEG hash,
- app share buttons use the same manifest/helper-derived path as the generated card.

Add focused tests for the share helper itself so path/image/page URL formatting is deterministic from token ID plus manifest version. Add a focused generator test or share-preview assertion proving manifest versions match the short hash of the generated JPEG bytes.

## Data flow

```text
STATIC_DEMO_DATA.agentCards
  -> numeric share catalog
  -> generated SVG
  -> generated PNG/JPEG preview images
  -> JPEG hash versions
  -> generated share-card manifest
  -> shared URL helper
  -> generated crawler HTML + app share button URL
```

The same generated manifest entry drives both the static crawler page and the app share URL, so they cannot drift independently.

## Error handling

- Skip non-numeric cards such as collection/swarm cards until they have an explicit share route design.
- If an image URL cannot be fetched, render a deterministic initials fallback and include a warning in generator output.
- If `ffmpeg`, `svg_pipe`, the `svg` decoder, or image rendering fails, exit non-zero.
- Generate all artifacts through temporary paths in `apps/web/public/share/.generated-tmp/` or equivalent, then replace final HTML, PNG, JPEG, and manifest outputs only after all cards render and manifest versions are known. This prevents mixed old/new card sets after a partial failure.
- If a token ID has no generated manifest entry, return `null` from share helpers and omit the static share button instead of linking to a missing crawler route.
- Do not delete orphaned old share files in this slice. If a numeric static card is removed later, stale files can be removed in an explicit cleanup change after verifying no live shares rely on them.
- Use a bot-detection JavaScript redirect only, matching the current static pages. Do not use HTTP redirects or meta refresh because crawlers need to read the metadata page.
- If a generated file would be stale relative to `STATIC_DEMO_DATA`, tests should fail.
- Keep all crawler metadata public-safe. No private headers, cookies, wallet data, or manager state.

## Testing and verification

Minimum gate:

```bash
pnpm generate:share-cards
pnpm generate:share-cards --check
node --test apps/web/test/share-cards.test.mjs apps/web/test/share-preview.test.mjs apps/web/test/app.test.mjs --test-name-pattern "share|Quigbot"
pnpm test
pnpm web:build
git diff --check
git diff --exit-code -- apps/web/public/share apps/web/src/generated-share-cards.js
```

Live deploy verification after merge:

```bash
node --input-type=module <<'NODE'
import assert from 'node:assert/strict';
import { createHash } from 'node:crypto';
import { GENERATED_SHARE_CARDS } from './apps/web/src/generated-share-cards.js';

const card = GENERATED_SHARE_CARDS['81'];
assert.ok(card?.version, 'missing generated share-card version for token 81');

const pageUrl = `https://helixa.xyz/multipass/share/81/?v=${encodeURIComponent(card.version)}`;
const imageUrl = `https://helixa.xyz/multipass/share/81.jpg?v=${encodeURIComponent(card.version)}`;
const headers = { 'user-agent': 'TelegramBot (like TwitterBot)' };

const page = await fetch(pageUrl, { headers });
assert.equal(page.status, 200);
assert.match(page.headers.get('content-type') ?? '', /text\/html/);
const html = await page.text();
assert.match(html, new RegExp(`<meta property="og:image" content="${imageUrl.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')}" \/>`));
assert.match(html, new RegExp(`<meta name="twitter:image" content="${imageUrl.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')}" \/>`));
assert.match(html, /<meta name="multipass:visual-source" content="https:\/\/assets\.bueno\.art\/images\/3b04f823-b7a8-4965-b61e-8fe8a5d82bde\/default\/2432" \/>/);

const image = await fetch(imageUrl, { headers });
assert.equal(image.status, 200);
assert.match(image.headers.get('content-type') ?? '', /image\/jpeg/);
const bytes = Buffer.from(await image.arrayBuffer());
const hashPrefix = createHash('sha256').update(bytes).digest('hex').slice(0, card.version.length);
assert.equal(hashPrefix, card.version);
console.log({ pageUrl, imageUrl, bytes: bytes.length, hashPrefix });
NODE
```

## Rollout

This is a source-only Multipass web update until deployed. It does not require restarting `multipass-api.service` unless unrelated API code changes are added.

Before deploy, back up `/var/www/helixa.xyz/multipass/`, build the web bundle, rsync `apps/web/dist/` to the live Multipass path, and smoke-test the versioned share route.
